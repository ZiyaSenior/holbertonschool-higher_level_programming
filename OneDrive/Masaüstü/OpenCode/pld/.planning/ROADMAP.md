# Roadmap: Student Dashboard UI Refresh

## Overview

This roadmap keeps the work scoped to a focused brownfield redesign of the existing student dashboard. It starts by correcting the page hierarchy and primary actions, then hardens the sessions workflow, reshapes announcements and PLD preferences into calmer secondary surfaces, and finishes with the preference reliability and responsive polish needed for launch.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions if later needed

- [ ] **Phase 1: Dashboard Foundation and Next-Action Hierarchy** - Reorder the page around one clear next action and remove stats-first clutter.
- [ ] **Phase 2: Sessions-First Workflow** - Make sessions the clearest day-to-day workflow with direct join actions and consistent state handling.
- [ ] **Phase 3: Calm Secondary Surfaces** - Rebuild announcements and PLD preferences as compact supporting sections that do not compete with primary actions.
- [ ] **Phase 4: Preference Reliability and Responsive Hardening** - Repair the preference save path and validate the redesigned dashboard across desktop and mobile.

## Phase Details

### Phase 1: Dashboard Foundation and Next-Action Hierarchy
**Goal:** Establish a student-first dashboard structure with one top next action, a persistent practice fallback, and calm empty states.
**Depends on:** Nothing (first phase)
**Requirements:** [DASH-01, DASH-02, DASH-03, PRAC-01]
**Success Criteria** (what must be TRUE):
  1. Student lands on the dashboard and sees one clear top-of-page next action before any secondary content.
  2. Student can start AI practice from a persistent action near the top of the page when no session needs immediate attention.
  3. Student can scan the page without seeing a stats-first or admin-style summary strip above the primary actions.
  4. Student sees calm empty states that point to a useful next step instead of dead-end placeholders.

### Phase 2: Sessions-First Workflow
**Goal:** Deliver a sessions-first workflow with clear ranking, direct join actions, and one consistent session time presentation.
**Depends on:** Phase 1
**Requirements:** [SESS-01, SESS-02, SESS-03, SESS-04]
**Success Criteria** (what must be TRUE):
  1. Student sees the most relevant session first with a clear topic, date, and start time.
  2. Student can distinguish join-now, upcoming, and other joinable session states from the dashboard without guessing.
  3. Student can join an eligible future session directly from the session area on the dashboard.
  4. Student sees the same understandable time format wherever session times appear on the dashboard.

### Phase 3: Calm Secondary Surfaces
**Goal:** Give announcements and PLD preferences a calm, compact presentation that supports scanning without competing with sessions and practice.
**Depends on:** Phase 2
**Requirements:** [ANNC-01, ANNC-02, PREF-01, PREF-03]
**Success Criteria** (what must be TRUE):
  1. Student can scan a compact list of the latest announcements with short previews and open the full announcements page.
  2. Student sees the current PLD preference as a compact summary instead of a dominant dashboard panel.
  3. Student can clearly review available PLD start times when looking at or entering the preference flow.
  4. Student experiences a calm, simple visual hierarchy where sessions and practice remain primary and utilities stay visually secondary.

### Phase 4: Preference Reliability and Responsive Hardening
**Goal:** Repair the PLD preference interaction path, confirm save feedback and state reconciliation, and harden the redesigned dashboard for desktop and mobile use.
**Depends on:** Phase 3
**Requirements:** [PREF-02, PREF-04, RESP-01]
**Success Criteria** (what must be TRUE):
  1. Student can edit their PLD preference from the dashboard without hitting the current error path.
  2. Student receives clear success or failure feedback after saving a PLD preference.
  3. Student sees the updated PLD preference reflected in the compact dashboard state after a successful save.
  4. Student can complete the primary dashboard actions on desktop and mobile without hidden CTAs or broken reading order.
